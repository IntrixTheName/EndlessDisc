Full-Stack Project Idea: Music Library Manager

I have a USB drive that contains 3000+ tracks and a portable music player for portable music needs. Mp3’s allow for a wide range of tags to be applied, but are too narrow of categories to be used in a specific situation like the one surrounding the Music Drive. Since the tags are limited and different stakeholders require different tag standards, a database with a front-end program can grab different tags for which standard is needed.

Use Case/Context

The file structure and tags attached to an Mp3 file are handled differently depending on the music player hosting the library. For example, my car has 2 specific limitations on local music files- No directory can exceed 128 items (directories or files) and there is only 20Gb of available space (embedded pictures & lyrics are effectively wasted space). Compare that to the music player on my phone which has no ability to subcategorize an album based on a Disc# tag or similar, so some of the albums on the Music Drive may not function properly.

Since all of these devices have specific needs that contradict with each other, this project aims to maintain a singular master copy of the music library, then deliver tailored copies of the library to meet the format of different devices on demand.

Project Overview

This project will consist of “reconstructing” an Mp3 file from raw audio data (likely aac) and metadata information stored in a database. There will then be either a C++ script/similar or an external program that can reconstruct the file. The resulting files will be placed into a .zip or other archive format and delivered to the user.

Project Specifics

The following sections detail how individual components of the project are envisioned

Database

The database will contain five tables, see below. The metadata table will contain all possible information about the music, and has been trimmed for this document. The user & configuration tables reflect the users that have a stake in this library and preset configurations that the user can select from. The track & cover art tables relate the internal ID’s to a file name/location for retrieval.

Metadata Table

The following list of columns is anticipated. Note that some columns share a name with a standard ID3v2.3+ tags, but are not necessarily related:

Internal Information: Track ID, Cover Art ID

Track Information: Title, Album, Sub-Album/Disc#, Track#, Artist, Year, Grouping, Language, Lyrics

Stakeholder Information*: Owner(s), Listener(s), Favor

* Stakeholder information intended to separate tracks based on stakeholder preferences. Ex, “owner” signifies which stakeholder contributed a track to the shared library, “listeners”/“favor” signifies which stakeholder(s) like/listen to the track. These can be used as fields to determine a subsection of the total library to download without manually selecting 1000’s of tracks by hand.

User & Configuration Tables

The following tables manage users & configurations. This helps to save presets so the information doesn’t have to be re-entered every time a request is made.

User Table: User ID, Username, Password/other entrance criteria, Configuration Scheme ID(s)

Configuration Table*: Configuration Scheme ID, Scheme Name, Tag Structure Info, File Structure Info

* Tag structure can be stored as plain text, something similar to |Info|-|Tag|\n|Inf..., or another standardized format. Similar for File Structure, which can be described similar to /Album/Disc#/ (describes directories created per album, subdirectories by Disc# tag before reaching the file).

Track & Cover Art Tables

The following tables link the internal tags to a file location for retrieval in the file reconstruction. Since these files cannot easily exist in the database itself, it may be easier to create a reference and link to the file. This can also help with file duplication*

Track Table: Track ID, File Path

Cover Art Table: Cover Art ID, File Path

* For example, 「Above the Waves」 is found in albums 「Let’s Cast Off」 and 「Faking Smiles」. They will have separate rows in the Metadata table, but will both link to the same physical file as the audio is the same for both. Additionally, it is possible (though no current examples exist) that multiple albums may share the same album art.

User Interface

The user interface is where users will log in, select a preset, and have the files delivered. The following sections overview how components of this will work.

Login: The login contains a username and some form of password, passphrase, pin, or other authentication piece to access the library. As there isn’t sensitive information attached with the information in the database, and the only sensitive information might be the login itself, this login does not need many security features, and is mainly used just to identify the stakeholder and maintain custom configurations.

Configurations: When a user logs in, they should have a way to create, load, modify, and remove custom configurations associated with their login. There could potentially also be a section dedicated to a one-time configuration creator. These configurations determine the tags that are transferred to the file, and determine the file structure that the library will be delivered in.

Library Selection: Based two potential methods of separating the library- Manual selection, and the previously mentioned Stakeholder Information columns. By selecting a subset of the library, this can further tailor the entire library to the user and help alleviate additional work/storage space on the client’s device. For example, I might only want songs I listen to regularly on my car since I have a limited space to work with.

File Construction

When a file and tag selection is made, the server side will construct the files based on the raw audio data and the tags selected. Given that an Mp3 file is mainly a container of metadata surrounding a raw audio file like .aac or .flac, constructing an Mp3 is relatively simple and only involves building a header block with the selected information and copy-pasting the raw audio data after the header. This can be done with something like a C++ script (C++ or similar needed because the header block cares about individual bits & bytes, so I need the lower level control of the file). Each of the tags is added by including a tag identifier (a four-letter code that identifies the piece of information being included), the length in bytes for the remainder of the tag, the encoding of the information (ASCII, Unicode, PNG/JPG, etc.), and the information of the tag. Once the tags are assembled, the files can be organized into a specified directory structure before being zipped and sent to the user.

Additional Notes

  • It may be easier to use an external command line-based tool to do the modifications of the tags rather than trying to build another tool to do that in-house. This option will be considered depending on what OS is running on Jordan’s proposed server and what tools may be associated with that.
  • Since there is no sensitive data contained as part of this project, I don’t believe that much is needed in the way of security, and thus I may not be following the best security practices in this project depending on how much time I have to implement it.

Sample Requirements/User Stories

  • The project should maintain multiple configurations per user
  • The project should be able to make a sub-selection so we can only select the music we want to download
  • The project should make configurations and the music library highly customizable, and take future expansion into account
